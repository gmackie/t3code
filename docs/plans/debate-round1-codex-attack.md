1. The proposal is solving a hypothetical scaling problem, not a demonstrated one. It cites one concrete browser PR, then jumps to imagined future panels, imagined contributor pressure, and an imagined “5th contributor” bottleneck in [docs/plans/2026-03-21-extension-host-proposal.md:11](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L11) and [docs/plans/2026-03-21-extension-host-proposal.md:17](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L17). That is YAGNI. You do not build a plugin platform because you can picture one someday.

2. The headline claim is already false. “Zero changes to core layout code” and “1 file + 1 registry line” in [docs/plans/2026-03-21-extension-host-proposal.md:5](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L5), [docs/plans/2026-03-21-extension-host-proposal.md:42](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L42), and [docs/plans/2026-03-21-extension-host-proposal.md:128](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L128) does not match the actual app structure. Panel identity is still hardcoded in [apps/web/src/rightPanelStateStore.ts:5](/Volumes/dev/t3code/apps/web/src/rightPanelStateStore.ts#L5), [apps/web/src/components/chat/ChatHeader.tsx:30](/Volumes/dev/t3code/apps/web/src/components/chat/ChatHeader.tsx#L30), and the route logic in [apps/web/src/routes/_chat.$threadId.tsx:248](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L248), [apps/web/src/routes/_chat.$threadId.tsx:429](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L429), and [apps/web/src/routes/_chat.$threadId.tsx:680](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L680). Any panel that wants persistence, shortcuts, mobile behavior, or URL state still touches core.

3. The “stable contract” is fiction. The proposal says extensions only need `ExtensionContext` and do not need to understand Zustand or transport internals in [docs/plans/2026-03-21-extension-host-proposal.md:60](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L60). The real browser and preview extensions immediately reach into app internals: [apps/web/src/extensions/builtins/browserExtension.tsx:23](/Volumes/dev/t3code/apps/web/src/extensions/builtins/browserExtension.tsx#L23) and [apps/web/src/extensions/builtins/previewWorkspace.tsx:17](/Volumes/dev/t3code/apps/web/src/extensions/builtins/previewWorkspace.tsx#L17). That means the real API is not `@t3tools/extension-api`; it is “whatever private web-app modules your extension happens to import.”

4. The package split is ceremony without isolation. The proposal treats five new packages as a minor cost in [docs/plans/2026-03-21-extension-host-proposal.md:121](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L121), but `ext-browser` and `ext-preview-workspace` are currently metadata stubs, not real standalone extensions, as admitted in [packages/ext-browser/src/index.ts:4](/Volumes/dev/t3code/packages/ext-browser/src/index.ts#L4) and [packages/ext-preview-workspace/src/index.ts:4](/Volumes/dev/t3code/packages/ext-preview-workspace/src/index.ts#L4). They are also private workspace packages in [packages/ext-browser/package.json:4](/Volumes/dev/t3code/packages/ext-browser/package.json#L4) and [packages/ext-preview-workspace/package.json:4](/Volumes/dev/t3code/packages/ext-preview-workspace/package.json#L4). This adds manifests, tsconfigs, dependency edges, and review overhead without buying actual decoupling.

5. This creates multiple panel models instead of one clear one. Diff stays special and URL-routed in [docs/plans/2026-03-21-extension-host-proposal.md:36](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L36), browser is special in the persisted panel store and header toggles in [apps/web/src/rightPanelStateStore.ts:5](/Volumes/dev/t3code/apps/web/src/rightPanelStateStore.ts#L5) and [apps/web/src/components/chat/ChatHeader.tsx:99](/Volumes/dev/t3code/apps/web/src/components/chat/ChatHeader.tsx#L99), and the API already invents future surfaces like `thread.headerActions` and `project.toolView` in [packages/extension-api/src/index.ts:109](/Volumes/dev/t3code/packages/extension-api/src/index.ts#L109). That is not simplification. That is contributor confusion encoded as architecture.

6. The claimed benefits are mostly theoretical. “Independently shippable” in [docs/plans/2026-03-21-extension-host-proposal.md:130](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L130) is not real when registration is a static compile-time list in [apps/web/src/extensions/builtinRegistry.ts:6](/Volumes/dev/t3code/apps/web/src/extensions/builtinRegistry.ts#L6), the packages are private, and there is no versioning, dynamic loading, compatibility policy, styling boundary, permissions model, or failure isolation. This is not a plugin system. It is a registry array.

7. The host design is underpowered and regresses UX predictability. `ExtensionHost` stores the selected extension in local component state and falls back to the first available one on remount in [apps/web/src/extensions/ExtensionHost.tsx:15](/Volumes/dev/t3code/apps/web/src/extensions/ExtensionHost.tsx#L15). There is no per-thread persistence, no URL representation, no inactive-tab state retention, and no error boundary around `activeExtension.render(...)` in [apps/web/src/extensions/ExtensionHost.tsx:42](/Volumes/dev/t3code/apps/web/src/extensions/ExtensionHost.tsx#L42). For a project that claims reliability and predictability under reconnects, that is a weak foundation.

8. The abstraction is not actually absorbing complexity. The browser extension still owns subscriptions, native API calls, tab lifecycle, viewport sync, and browser state coordination in [apps/web/src/extensions/builtins/browserExtension.tsx:62](/Volumes/dev/t3code/apps/web/src/extensions/builtins/browserExtension.tsx#L62) through [apps/web/src/extensions/builtins/browserExtension.tsx:279](/Volumes/dev/t3code/apps/web/src/extensions/builtins/browserExtension.tsx#L279). So the hard part did not disappear; it just moved into another file and gained another indirection layer. That is not simplification. That is redistribution.

9. The performance story is weak and already shows premature cleverness. `selectExtensionThreadView` uses a module-global single-entry cache in [apps/web/src/extensions/extensionSelectors.ts:16](/Volumes/dev/t3code/apps/web/src/extensions/extensionSelectors.ts#L16), which is a brittle micro-optimization, not a robust memoization strategy. At the same time, the extension model encourages more `isAvailable` checks, more wrapper renders, and more side-effectful extension components without a disciplined lifecycle model.

10. The testing evidence is padded. The proposal leans on total repo test counts and review counts in [docs/plans/2026-03-21-extension-host-proposal.md:112](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L112), not on meaningful coverage of the extension mechanism itself. The extension-specific tests I found are limited to [apps/web/src/extensions/registry.test.ts](/Volumes/dev/t3code/apps/web/src/extensions/registry.test.ts), [apps/web/src/extensions/extensionSelectors.test.ts](/Volumes/dev/t3code/apps/web/src/extensions/extensionSelectors.test.ts), and [apps/web/src/extensions/builtins/planningWorkbench.logic.test.ts](/Volumes/dev/t3code/apps/web/src/extensions/builtins/planningWorkbench.logic.test.ts). There is no evidence here that host rendering, tab switching, browser lifecycle, preview persistence, mobile layout, or extension failure behavior are actually stable.

11. There is a much simpler alternative, which is why this proposal is hard to justify. If the real problem is right-panel sprawl, extract a local `RightPanelController` and shared right-panel plumbing inside `apps/web` first. Jumping straight to a public extension API, five packages, future repo boundaries, and unused extension surfaces is premature abstraction with permanent maintenance cost.

I would reject this as written. It is mostly future-tense architecture, fake boundaries, and extra moving parts around a problem that has not been proven to need a platform.
