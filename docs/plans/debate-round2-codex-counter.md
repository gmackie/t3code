**Concessions**
- Dropping the stub packages is not cosmetic; it blows up the proposal’s main decoupling story. The proposal sells package and eventual repo boundaries as core evidence of the design [proposal:64](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L64), [proposal:75](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L75). The rebuttal then admits `ext-browser` and `ext-preview-workspace` are just metadata stubs [rebuttal:7](/Volumes/dev/t3code/docs/plans/debate-round1-claude-rebuttal.md#L7), and those packages explicitly say the real implementation stays in `apps/web` [ext-browser:4](/Volumes/dev/t3code/packages/ext-browser/src/index.ts#L4), [ext-preview:4](/Volumes/dev/t3code/packages/ext-preview-workspace/src/index.ts#L4). That is not “directory structure first”; it is proof the hard cases do not fit the contract.
- The testing concession reveals the same problem. The proposal leans on total repo test counts as evidence [proposal:112](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L112), then the rebuttal admits the host is missing component coverage for tab switching, availability changes, and responsive behavior [rebuttal:11](/Volumes/dev/t3code/docs/plans/debate-round1-claude-rebuttal.md#L11). For a project that prioritizes predictable behavior under reconnects and UI transitions, that is not a small POC gap; it means the new abstraction was proposed before its lifecycle was proven.

**Weak Defenses**
- The “planning workbench and thread overview prove the clean contract” defense is factually wrong. The rebuttal says those extensions use only `ExtensionContext` [rebuttal:33](/Volumes/dev/t3code/docs/plans/debate-round1-claude-rebuttal.md#L33). `threadOverview.tsx` imports app UI directly [threadOverview:1](/Volumes/dev/t3code/apps/web/src/extensions/builtins/threadOverview.tsx#L1), and `planningWorkbench.tsx` imports app UI, hooks, `readNativeApi`, and `toastManager` [planning:3](/Volumes/dev/t3code/apps/web/src/extensions/builtins/planningWorkbench.tsx#L3), [planning:8](/Volumes/dev/t3code/apps/web/src/extensions/builtins/planningWorkbench.tsx#L8), [planning:9](/Volumes/dev/t3code/apps/web/src/extensions/builtins/planningWorkbench.tsx#L9), [planning:11](/Volumes/dev/t3code/apps/web/src/extensions/builtins/planningWorkbench.tsx#L11). The only code actually extracted to `@t3tools/ext-planning-workbench` is string-processing logic [ext-planning:1](/Volumes/dev/t3code/packages/ext-planning-workbench/src/index.ts#L1), [planning-logic:4](/Volumes/dev/t3code/apps/web/src/extensions/builtins/planningWorkbench.logic.ts#L4).
- The “we already have multiple panels” argument is circular. The rebuttal counts diff, plan sidebar, browser, overview, and terminal as proof the host is warranted [rebuttal:17](/Volumes/dev/t3code/docs/plans/debate-round1-claude-rebuttal.md#L17). But the proposal itself keeps diff special [proposal:36](/Volumes/dev/t3code/docs/plans/2026-03-21-extension-host-proposal.md#L36), the current app keeps the plan sidebar separate [ChatView:4175](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L4175), and terminal is a drawer with its own persisted store [ChatView:4202](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L4202), [terminalStore:2](/Volumes/dev/t3code/apps/web/src/terminalStateStore.ts#L2). That argues for explicit local surface modeling, not a public extension API.
- The “host is a `RightPanelController` plus contract, availability filtering, and tab management” line is weak. Availability filtering is just `filter(...isAvailable)` [registry:8](/Volumes/dev/t3code/apps/web/src/extensions/registry.ts#L8), and tab management is a local `useState` with fallback-to-first [host:19](/Volumes/dev/t3code/apps/web/src/extensions/ExtensionHost.tsx#L19), [host:28](/Volumes/dev/t3code/apps/web/src/extensions/ExtensionHost.tsx#L28). Open/close, mutual exclusion, routing, persistence, mobile sheet vs desktop sidebar, and browser lifecycle are all still elsewhere [ChatView:347](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L347), [ChatView:1611](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L1611), [_chat.$threadId:248](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L248), [_chat.$threadId:429](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L429), [_chat.$threadId:732](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L732).

**Why `RightPanelController` Is Still Better**
- It attacks the real coupling points in this codebase: per-thread right-panel persistence [rightPanelStore:5](/Volumes/dev/t3code/apps/web/src/rightPanelStateStore.ts#L5), [rightPanelStore:83](/Volumes/dev/t3code/apps/web/src/rightPanelStateStore.ts#L83), header toggles [ChatHeader:30](/Volumes/dev/t3code/apps/web/src/components/chat/ChatHeader.tsx#L30), [ChatHeader:99](/Volumes/dev/t3code/apps/web/src/components/chat/ChatHeader.tsx#L99), keyboard shortcuts [ChatView:1176](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L1176), [ChatView:2214](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L2214), diff URL state [_chat.$threadId:85](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L85), browser lifecycle [_chat.$threadId:277](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L277), and responsive shell selection [_chat.$threadId:732](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L732). The extension host does not unify those; it sits beside them.
- It keeps the API honest. A local controller can start with internal panel descriptors and only extract a public contract once reuse is real. The current proposal publishes unused surfaces in the “stable” API [extension-api:109](/Volumes/dev/t3code/packages/extension-api/src/index.ts#L109), even though only `thread.sidePanel` is hosted [host:16](/Volumes/dev/t3code/apps/web/src/extensions/ExtensionHost.tsx#L16) and `thread.headerActions` exists only in a test [registry-test:48](/Volumes/dev/t3code/apps/web/src/extensions/registry.test.ts#L48).
- It is lower-cost and actually simplifies state. Today the extension path adds `extensionPanelOpen` and `planSidebarOpen` in `ChatView` [ChatView:346](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L346), while `selectedPanel` remains a separate per-thread model in the route [_chat.$threadId:248](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L248). A `RightPanelController` would collapse those controls into one place instead of adding a second sidebar framework.

**New Weaknesses Exposed By The Rebuttal**
- The public contract already lies. `ExtensionContext.openSidePanel` takes a `panelId` [extension-api:132](/Volumes/dev/t3code/packages/extension-api/src/index.ts#L132), but the implementation ignores it and just flips a boolean [ChatView:768](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L768). `actions` are part of the contract [extension-api:122](/Volumes/dev/t3code/packages/extension-api/src/index.ts#L122), [extension-api:134](/Volumes/dev/t3code/packages/extension-api/src/index.ts#L134), but the app always passes `[]` [ChatView:773](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L773).
- Common concerns still leak out of the host. Preview URL cleanup is wired in the app root [__root:188](/Volumes/dev/t3code/apps/web/src/routes/__root.tsx#L188), browser cleanup spans root and route [__root:183](/Volumes/dev/t3code/apps/web/src/routes/__root.tsx#L183), [_chat.$threadId:402](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L402), and the browser extension still owns native subscriptions and host sync [browserExt:62](/Volumes/dev/t3code/apps/web/src/extensions/builtins/browserExtension.tsx#L62), [browserExt:100](/Volumes/dev/t3code/apps/web/src/extensions/builtins/browserExtension.tsx#L100). So the host is not centralizing lifecycle; it is distributing it.
- The rebuttal says the host reduces the app to two panel models [rebuttal:41](/Volumes/dev/t3code/docs/plans/debate-round1-claude-rebuttal.md#L41). The current implementation has at least three: route-owned per-thread diff/browser state [rightPanelStore:83](/Volumes/dev/t3code/apps/web/src/rightPanelStateStore.ts#L83), route-enforced diff URL state [_chat.$threadId:85](/Volumes/dev/t3code/apps/web/src/routes/_chat.$threadId.tsx#L85), and `ChatView`’s local extension/plan sidebar booleans [ChatView:346](/Volumes/dev/t3code/apps/web/src/components/ChatView.tsx#L346). If terminal counts, it is four [terminalStore:2](/Volumes/dev/t3code/apps/web/src/terminalStateStore.ts#L2).

The rebuttal narrowed the claims, but that made the underlying problem clearer: this is not a proven extension architecture. It is an internal extension sidebar layered on top of an already-fragmented panel system, with a public API that is broader than the implementation and cleaner than the real dependencies.
